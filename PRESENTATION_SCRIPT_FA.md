# متن ارائه فارسی پروژه پویا

## معرفی کوتاه یک‌دقیقه‌ای

سلام استاد. پروژه‌ای که ارائه می‌دهم «پویا» نام دارد و یک سامانه مدیریت رستوران است. این سیستم برای ثبت سفارش، مدیریت منو، مدیریت موجودی روزانه، نمایش وضعیت سفارش‌ها برای آشپزخانه و مشتری، چاپ فاکتور و تحلیل فروش طراحی شده است.

از نظر فنی، Backend با Django REST Framework نوشته شده و Frontend با Vue 3 و Vite ساخته شده است. همچنین برای اجرای دسکتاپ از PyWebView استفاده شده تا برنامه شبیه یک نرم‌افزار ویندوزی اجرا شود و بتواند به قابلیت‌هایی مثل چاپگر حرارتی دسترسی داشته باشد.

## متن کامل ارائه ۵ تا ۷ دقیقه‌ای

سلام استاد. من امروز پروژه «پویا» را معرفی می‌کنم. این پروژه یک سیستم مدیریت رستوران فارسی است که هدفش ساده‌تر کردن فرآیند فروش و آماده‌سازی سفارش‌ها در رستوران است.

در این سیستم، کاربر ابتدا وارد صفحه ورود می‌شود. بعد از ورود، بسته به نقشی که دارد، وارد بخش مناسب می‌شود. مثلاً مدیر می‌تواند صفحه فروش و آمار را ببیند، آشپز وارد برد سفارش‌های آشپزخانه می‌شود، صندوقدار سفارش ثبت می‌کند و مشتری یا نقش نمایشی می‌تواند وضعیت سفارش‌ها را مشاهده کند.

معماری پروژه به‌صورت client/server است. سمت سرور با Django و Django REST Framework پیاده‌سازی شده است. این قسمت مدل‌های اصلی مثل غذا، دسته‌بندی، موجودی روزانه، میز، سفارش و آیتم سفارش را مدیریت می‌کند. سمت کاربر با Vue 3 ساخته شده و از Axios برای ارتباط با API استفاده می‌کند. مسیرهای frontend با Vue Router مدیریت می‌شوند و برای ظاهر برنامه از Element Plus و فونت Vazir استفاده شده تا رابط کاربری فارسی و راست‌به‌چپ باشد.

جریان اصلی برنامه این است که صندوقدار یا کاربر مجاز وارد Dashboard می‌شود، لیست غذاها را همراه با موجودی امروز می‌بیند، آیتم‌ها را به سفارش اضافه می‌کند، اطلاعات مشتری و میز را وارد می‌کند و سفارش را ثبت می‌کند. بعد از ثبت سفارش، داده به Backend ارسال می‌شود و در پایگاه داده ذخیره می‌گردد. سپس آشپز در صفحه KitchenOrders سفارش‌های امروز را می‌بیند و وضعیت آن‌ها را از «در انتظار» به «در حال آماده‌سازی»، بعد «آماده» و در نهایت «تکمیل‌شده» تغییر می‌دهد.

یکی از بخش‌های کاربردی پروژه، صفحه وضعیت سفارش‌هاست. این صفحه فقط اطلاعات ضروری سفارش‌های امروز را نشان می‌دهد تا مشتری یا کارکنان بتوانند ببینند سفارش در چه مرحله‌ای قرار دارد. برای مدیریت بهتر رستوران، صفحه Statistics هم وجود دارد که آمار فروش، درآمد، سفارش‌ها، غذاهای پرفروش، فعالیت میزها و حتی پیش‌بینی ساده فروش و تقاضا را نمایش می‌دهد.

از نظر Backend، routeهای اصلی در فایل URL پروژه ثبت شده‌اند و برای مدل‌های اصلی از ViewSetهای Django REST Framework استفاده شده است. احراز هویت با Token انجام می‌شود؛ یعنی کاربر بعد از login یک token دریافت می‌کند و frontend این token را در درخواست‌های بعدی داخل header می‌فرستد. مجوزها هم بر اساس گروه‌های Django مثل admin، chef، cashier و customer تعریف شده‌اند.

یک نکته دیگر این است که پروژه فقط web نیست؛ با PyWebView می‌تواند به‌صورت یک برنامه دسکتاپ اجرا شود. PyWebView فایل build‌شده frontend را از پوشه dist سرو می‌کند و یک window دسکتاپ باز می‌کند. همچنین از طریق یک API پایتونی، frontend می‌تواند درخواست چاپ فاکتور را به چاپگر حرارتی ارسال کند. اگر PyWebView در دسترس نباشد، سرویس چاپ از browser print به‌عنوان fallback استفاده می‌کند.

پایگاه داده فعلی SQLite است. برای یک پروژه آموزشی یا محیط داخلی کوچک مناسب است، ولی اگر قرار باشد پروژه در production بزرگ استفاده شود، بهتر است دیتابیس به PostgreSQL یا یک دیتابیس مناسب‌تر تغییر کند. همچنین بعضی تنظیمات مثل SECRET_KEY، DEBUG و CORS باید از حالت توسعه خارج شوند و با environment variable کنترل شوند.

در مجموع، این پروژه یک نمونه نسبتاً کامل از ترکیب Backend REST، Frontend مدرن، نقش‌های کاربری، مدیریت سفارش و اجرای دسکتاپ است. ارزش پروژه در این است که یک مسئله واقعی، یعنی مدیریت سفارش رستوران، را با معماری قابل فهم و قابل توسعه پیاده‌سازی می‌کند.

## توضیح معماری به زبان ساده

این پروژه سه لایه اصلی دارد:

1. **Frontend**: همان چیزی است که کاربر می‌بیند؛ صفحات ورود، فروش، آشپزخانه، وضعیت سفارش و آمار.
2. **Backend/API**: مغز اصلی سیستم است؛ داده‌ها را ذخیره می‌کند، سفارش می‌سازد، نقش‌ها را بررسی می‌کند و آمار تولید می‌کند.
3. **Desktop/PyWebView**: یک پوسته دسکتاپ است که frontend را داخل پنجره برنامه اجرا می‌کند و امکان چاپ را به برنامه می‌دهد.

به زبان ساده، Vue فرم‌ها و صفحات را نشان می‌دهد، Django داده‌ها و قوانین را نگه می‌دارد، و PyWebView برنامه را شبیه نرم‌افزار دسکتاپ اجرا می‌کند.

## توضیح main code flow

1. برنامه frontend از `src/main.js` شروع می‌شود.
2. Router و Element Plus نصب می‌شوند.
3. `App.vue` فقط صفحه فعال را با `router-view` نشان می‌دهد.
4. اگر کاربر وارد نشده باشد، route guard او را به Login می‌فرستد.
5. Login اطلاعات را به Backend می‌فرستد و token می‌گیرد.
6. Dashboard با `api.menu.withInventory()` غذاها را می‌گیرد.
7. کاربر سفارش را ثبت می‌کند و frontend به `api.orders.create()` درخواست می‌فرستد.
8. Backend با `OrderCreateSerializer` سفارش و آیتم‌ها را ذخیره می‌کند.
9. KitchenOrders سفارش‌های امروز را با `api.orders.today()` می‌گیرد و وضعیت را با `update_status` تغییر می‌دهد.
10. Statistics از endpointهای analytics برای آمار و پیش‌بینی استفاده می‌کند.

## چرا این پروژه مفید است؟

- فرآیند ثبت سفارش رستوران را دیجیتال می‌کند.
- ارتباط بین صندوق و آشپزخانه را ساده‌تر می‌کند.
- وضعیت سفارش برای مشتری یا کارکنان قابل مشاهده است.
- امکان چاپ فاکتور دارد.
- مدیر می‌تواند عملکرد فروش را با نمودارها و پیش‌بینی‌ها بررسی کند.
- پروژه برای آموزش معماری full-stack بسیار مناسب است، چون Backend، Frontend، Auth، Role-based access، API و Desktop integration را با هم دارد.

## توضیح فنی برای سؤال‌های سطح استاد

### Backend چگونه طراحی شده است؟

Backend با Django REST Framework طراحی شده است. مدل‌های اصلی در اپ‌های جداگانه قرار دارند: `menu` برای غذا و موجودی، `orders` برای میز و سفارش، `users` برای احراز هویت و مجوزها، و `analytics` برای آمار. APIهای CRUD با ViewSet و DefaultRouter ساخته شده‌اند و endpointهای خاص مثل login و predictions با APIView پیاده‌سازی شده‌اند.

### احراز هویت چگونه کار می‌کند؟

کاربر username و password را ارسال می‌کند. اگر معتبر باشد، Django user را authenticate می‌کند و DRF token ساخته یا بازیابی می‌شود. سپس frontend این token را در localStorage ذخیره می‌کند و در هر درخواست بعدی آن را در header می‌فرستد.

### نقش‌ها چگونه کنترل می‌شوند؟

نقش‌ها با گروه‌های Django کنترل می‌شوند. permission classها بررسی می‌کنند که user عضو گروه admin، chef، cashier یا customer باشد. سمت frontend هم route guard دارد، اما کنترل اصلی امنیت باید در Backend باشد.

### آمار و پیش‌بینی چگونه تولید می‌شود؟

بخش analytics داده‌های سفارش و آیتم سفارش را جمع‌آوری می‌کند و با pandas آمارهایی مثل درآمد، تعداد سفارش، غذاهای پرفروش، سفارش بر اساس ساعت و فعالیت میزها را می‌سازد. پیش‌بینی‌ها ساده و آماری هستند؛ بر اساس میانگین روزهای هفته و میانگین متحرک اخیر.

### PyWebView چه نقشی دارد؟

PyWebView باعث می‌شود برنامه Vue داخل یک پنجره دسکتاپ اجرا شود. همچنین یک API پایتونی به frontend می‌دهد که برای چاپ فاکتور استفاده می‌شود. این یعنی frontend فقط یک درخواست JavaScript می‌زند، ولی چاپ واقعی از سمت Python انجام می‌شود.

## سؤال‌های احتمالی استاد و پاسخ‌های خوب

### سؤال: چرا از Django REST Framework استفاده شده؟

پاسخ: چون پروژه نیاز به API ساخت‌یافته برای چند entity مثل غذا، سفارش، میز و کاربر دارد. DRF با serializer، ViewSet، permission و authentication این کار را سریع و استاندارد انجام می‌دهد.

### سؤال: چرا Vue انتخاب شده؟

پاسخ: Vue برای ساخت رابط کاربری reactive و component-based مناسب است. در این پروژه، صفحه فروش، برد آشپزخانه و داشبورد آمار stateهای زیادی دارند و Vue مدیریت این stateها را ساده می‌کند.

### سؤال: آیا frontend route guard برای امنیت کافی است؟

پاسخ: خیر. route guard بیشتر برای تجربه کاربری است. امنیت واقعی در Backend و permission classها اعمال می‌شود، چون درخواست‌های API باید سمت سرور کنترل شوند.

### سؤال: چرا SQLite استفاده شده؟

پاسخ: برای توسعه، تست و محیط کوچک ساده است. اما برای production بهتر است از دیتابیس قوی‌تر مثل PostgreSQL استفاده شود.

### سؤال: نقطه ضعف اصلی پروژه چیست؟

پاسخ: تنظیمات فعلی بیشتر development هستند؛ مثلاً `DEBUG=True`، CORS باز و SECRET_KEY داخل کد قرار دارد. برای production باید این موارد با environment variables و تنظیمات امن جایگزین شوند.

### سؤال: اگر بخواهی پروژه را توسعه بدهی، چه کار می‌کنی؟

پاسخ: اول state management را با Pinia متمرکز می‌کنم، تست‌های Backend و Frontend اضافه می‌کنم، تنظیمات production را از env می‌خوانم، و مستندات API/deployment را کامل‌تر می‌کنم.

### سؤال: آیا این پروژه real-time است؟

پاسخ: در frontend برای بعضی صفحات مثل آشپزخانه polling استفاده شده است، یعنی هر چند ثانیه داده‌ها دوباره گرفته می‌شوند. در requirements کتابخانه Channels هم دیده می‌شود، اما پیاده‌سازی قطعی real-time WebSocket در مسیر اصلی پروژه به‌صورت کامل مشخص نیست.

## جمع‌بندی خیلی کوتاه

پروژه پویا یک سیستم مدیریت رستوران full-stack است که با Django REST Framework، Vue 3 و PyWebView ساخته شده است. این سیستم سفارش‌گیری، مدیریت آشپزخانه، نمایش وضعیت سفارش، چاپ فاکتور و آمار فروش را پوشش می‌دهد و برای یک پروژه دانشگاهی نمونه خوبی از معماری کاربردی client/server محسوب می‌شود.
